DOCUMENTACION DEL PROYECTO CRUD RAPIDO
====================================

1. Descripcion general
----------------------
Este proyecto es una aplicacion Spring Boot que gestiona estudiantes, parametros de horario y consume una API externa.
Incluye un CRUD de estudiantes con validaciones por grupos, paginacion, eliminacion logica, un filtro de horario, manejo global de respuestas y excepciones, un cliente WebClient moderno y un scheduler.

2. Tecnologias utilizadas
-------------------------
- Java 17
- Spring Boot 3.x
- Spring Web MVC
- Spring Data JPA
- Spring Validation
- Hibernate
- MySQL
- Lombok
- OpenAPI / Swagger
- WebClient / HttpServiceProxy
- Scheduling (@Scheduled)

3. Estructura del proyecto
--------------------------
src/main/java/com/example/crudrapido/
- CrudrapidoApplication.java
  Clase principal que inicia la aplicacion Spring Boot.
  Configura la propiedad java.net.preferIPv4Stack=true para evitar problemas de resolución DNS/IPv6 con WebClient.

- controller/
  - StudentController.java
    Expone los endpoints para listar, crear, actualizar, eliminar, obtener por ID y matricular estudiantes.
  - ParametrosController.java
    Expone el endpoint para actualizar la hora limite de acceso.
  - PruebaController.java
    Expone un endpoint para consumir posts desde la API externa JSONPlaceholder.

- service/
  - StudentService.java
    Logica de negocio para estudiantes, paginacion, creación, actualización, eliminacion y matricula en cursos.
  - ParametroService.java
    Logica para obtener y actualizar la hora limite almacenada en la tabla de parametros.
  - CourseService.java
    Logica para obtener cursos y soportar la matricula de estudiantes.

- entity/
  - Student.java
    Entidad que representa a un estudiante.
    Implementa eliminacion logica con @SQLDelete y aplica restriccion de consulta customizada.
    Mantiene relacion many-to-many con Course.
  - Course.java
    Entidad que representa un curso.
    Mantiene relacion many-to-many con Student.
  - Parametros.java
    Entidad que representa un parametro clave/valor.

- dto/
  - StudentDTO.java
    Objeto de transferencia para student con validaciones de creacion y actualizacion.
    Usa grupos @OnCreate y @OnUpdate para aplicar reglas distintas.
  - ParametroDTO.java
    DTO para recibir la nueva hora limite con formato HH:mm.
  - ApiResponse.java
    Estructura uniforme para envolver respuestas exitosas.
  - PostExternalDTO.java
    DTO para mapear la respuesta de la API externa JSONPlaceholder.

- repository/
  - StudentRepository.java
    Repositorio JPA para Student.
  - CourseRepository.java
    Repositorio JPA para Course.
  - ParametroRepository.java
    Repositorio JPA para Parametros.

- middlewares/
  - GlobalExceptionHandler.java
    Maneja excepciones de validacion y RuntimeException.
  - GlobalResponseHandler.java
    Envuelve en ApiResponse las respuestas exitosas, excepto ApiResponse, ResponseEntity, String y byte[].

- filter/
  - HorarioFilter.java
    Bloquea accesos a /api/* fuera de la hora limite configurada, excepto al endpoint /api/parametros.

- config/
  - FilterConfig.java
    Registra el filtro de horario sobre /api/* con orden 1.
  - WebClientConfig.java
    Configura WebClient y crea el cliente HttpServiceProxy para JSONPlaceholder.

- client/
  - JsonPlaceholderClient.java
    Interfaz declarativa que consume la API externa JSONPlaceholder.

- scheduler/
  - StudentScheduler.java
    Componente con @Scheduled que imprime un mensaje cada 10 minutos.

- validation/
  - EmailDominioValidator.java
    Valida que el correo pertenezca al dominio sasf.net.
  - ValidEmailDominio.java
    Anotacion personalizada para validar correos con dominio.

4. Funcionalidades principales
------------------------------
Modulo de estudiantes:
- Listar estudiantes con paginacion.
- Buscar un estudiante por ID.
- Crear un estudiante.
- Actualizar un estudiante.
- Eliminar un estudiante (logico) por ID.
- Matricular un estudiante en un curso.
- Validaciones para nombre, apellido, correo y ID.
- Transformacion de nombres y apellidos a mayusculas en creacion/actualizacion.
- Eliminacion logica mediante @SQLDelete en la entidad Student.

Modulo de cursos:
- Entidad Course con relacion many-to-many con Student.
- Matricula de estudiante en curso mediante POST /api/v1/students/{studentId}/courses/{courseId}.

Modulo de parametros:
- Actualizar la hora limite mediante POST /api/parametros.
- Lectura de la hora limite desde la base de datos.
- El horario limite controla el acceso a la mayoria de endpoints.

Modulo de seguridad y control de acceso:
- Filtro de horario que bloquea peticiones fuera del rango permitido.
- Excepciones y respuestas globales estandarizadas.

Modulo de consumo externo:
- Endpoint de prueba que obtiene posts desde JSONPlaceholder usando WebClient.

Modulo de programacion de tareas:
- Scheduler que ejecuta una tarea cada 600000 ms (10 minutos).

5. Endpoints principales
------------------------
Estudiantes:
- GET /api/v1/students?page=0&size=10
  Obtiene la lista de estudiantes paginada.
- GET /api/v1/students/{studentId}
  Obtiene un estudiante por su ID.
- POST /api/v1/students
  Crea un nuevo estudiante.
- PUT /api/v1/students
  Actualiza un estudiante existente.
- DELETE /api/v1/students/{studentId}
  Elimina un estudiante por ID (logico).
- POST /api/v1/students/{studentId}/courses/{courseId}
  Matricula un estudiante en un curso.

Parametros:
- POST /api/parametros
  Actualiza la hora limite de acceso.

Prueba externa:
- GET /api/prueba/posts
  Consume la API externa JSONPlaceholder y devuelve los posts.

6. Reglas de validacion
-----------------------
- studentId: debe ser mayor a 0 para actualizacion y eliminacion.
- firstName: entre 2 y 15 caracteres, solo letras y acentos en creacion.
- lastName: entre 2 y 15 caracteres, solo letras y acentos en creacion.
- email: obligatorio en creacion, formato valido y dominio sasf.net.
- nuevaHora: debe tener formato HH:mm.

7. Modelo de datos
------------------
Estudiante (Student):
- studentId: identificador unico.
- firstName: nombre.
- lastName: apellido.
- email: correo electronico unico.
- estado: estado del registro (ACTIVO, INACTIVO, N).
- courses: cursos asociados mediante relacion many-to-many.

Curso (Course):
- courseId: identificador unico.
- name: nombre del curso.
- description: descripcion del curso.
- students: estudiantes asociados.

Parametro (Parametros):
- clave: identificador del parametro.
- valor: valor asociado.

8. Notas importantes
--------------------
- GlobalResponseHandler normaliza respuestas exitosas en ApiResponse.
- GlobalExceptionHandler devuelve BAD_REQUEST para errores de validacion y NOT_FOUND para RuntimeException.
- HorarioFilter bloquea peticiones fuera del horario limite excepto el endpoint de parametros.
- La aplicacion usa MySQL y el esquema se actualiza automaticamente con spring.jpa.hibernate.ddl-auto=update.
- El puerto configurado es 8081.
- La propiedad java.net.preferIPv4Stack=true evita problemas de resolución DNS/IPv6 con WebClient.

9. Como ejecutar la aplicacion
-----------------------------
1. Configurar la base de datos MySQL en application.properties.
2. Ejecutar el comando:
   ./mvnw spring-boot:run
3. Acceder a la documentacion Swagger en:
   http://localhost:8081/swagger-ui/index.html

10. Resumen
-----------
Este proyecto es una API REST Spring Boot con gestion de estudiantes y cursos, validaciones personalizadas, eliminacion logica, filtro horario, manejo global de errores y respuestas, consumo de API externa via WebClient, y una tarea programada de ejemplo.

11. Seguridad y autenticación (JWT)
---------------------------------
Se añadió un módulo de autenticación basado en JWT para administradores. Puntos clave:

- Endpoints:
  - POST /api/v1/auth/login : recibe `LoginRequestDTO` (email, password) y devuelve `TokenResponseDTO` con `access_token` y `refresh_token`.

- DTOs y entidades:
  - `LoginRequestDTO` : contiene `email` y `password` (validaciones aplicadas).
  - `TokenResponseDTO` : devuelve `access_token` y `refresh_token` en el JSON.
  - `Token` (entidad): tabla `tbl_tokens` que guarda tokens de acceso emitidos, su estado (`revoked`, `expired`) y referencia a `Administrador`.

- Servicios y componentes:
  - `AuthService` : autentica con `AuthenticationManager`, genera access y refresh tokens con `JwtService`, revoca tokens previos y guarda el nuevo token en la BD.
  - `JwtService` : encapsula la generación y validación de JWT (generar token, extraer username, validar expiración). Requiere propiedades `jwt.secret`, `jwt.expiration` y `jwt.refresh.expiration` en `application.properties`.
  - `JwtAuthenticationFilter` : filtro `OncePerRequestFilter` que extrae el header `Authorization: Bearer ...`, valida el token con `JwtService` y con `TokenRepository` (revocado/expirado) y establece la autenticación en `SecurityContext`.

- Repositorio:
  - `TokenRepository` : consulta tokens válidos por usuario y permite buscar por token.

- Configuración de seguridad:
  - `SecurityConfig` : permite `/api/v1/auth/**` y rutas de Swagger públicamente; el resto requiere autenticación. Usa `JwtAuthenticationFilter` antes del `UsernamePasswordAuthenticationFilter` y mantiene la aplicación sin estado (`STATELESS`).
  - `AppConfig` : define `UserDetailsService` que busca `Administrador` por email, `AuthenticationProvider` con `DaoAuthenticationProvider` y el `PasswordEncoder` (`BCryptPasswordEncoder`).

- Recomendaciones y notas operativas:
  - `jwt.secret` debe ser una cadena Base64 con suficiente entropía (p. ej. mínimo 256 bits para HS256). Puedes generar una clave segura con `openssl rand -base64 32` o una herramienta equivalente.
  - Verificar que `jwt.expiration` y `jwt.refresh.expiration` estén expresados en ms o como está documentado en `application.properties`.
  - Manejar excepciones comunes de JWT (ExpiredJwtException, JwtException) si se desea respuestas más amigables.
  - El filtro `JwtAuthenticationFilter` omite rutas que contienen `/api/v1/auth` para permitir el inicio de sesión.
